# evals — Groundedness evaluation from Foundry traces, per agent & per model

Evaluate Foundry Prompt Agents for **groundedness** using their real production
traces, and compare **multiple models** on the same data. Results show up in the
**new Microsoft Foundry portal** under **Build → Evaluations** as one evaluation
with one run per model (a comparison table).

Traces are read **directly from Foundry's tracing store** — the Application
Insights / Log Analytics workspace wired to the Foundry project (the
`AppGenAIContent` table) — so no Cosmos or backend access is needed.

## Layout

```
evals/
  fetch_traces.py        # traces -> agents/<name>/dataset.jsonl
  run_eval.py            # dataset -> one eval, one run per model
  _common.py             # shared helpers (azd env, agent-folder resolution)
  requirements.txt
  agents/
    aurelia-motors/
      agent.json         # per-agent config (agent name, judge, models, ...)
      dataset.jsonl      # generated by fetch_traces.py
    <other-agent>/       # add more agents the same way
      agent.json
```

Each agent has its own folder under `agents/`. The shared scripts take the agent
folder name as their first argument, so the same tooling serves many agents.

## agent.json

```jsonc
{
  "agent": "knowledge-faq-assistant-aurelia-motors", // Foundry agent name (traces)
  "judge": "gpt-4o-mini",                             // LLM judge for groundedness
  "days": 90,                                          // trace look-back window
  "evaluation_name": "aurelia-groundedness",          // portal display name
  "models": ["gpt-4o-mini", "gpt-5.4", "grok-4.3"],   // models to compare
  "instructions": "You are a helpful FAQ assistant ..."// system prompt for targets
}
```

`models` are model **deployment names** in the Foundry project.

## How it works

Each agent turn sends the model a user message shaped like:

```
[Knowledge base context - answer using ONLY this when applicable]
- (source) title: content ...

[User question]
<the customer's question>
```

1. `fetch_traces.py` pulls those spans, splits each turn into **context** (the
   grounding knowledge) + **query** (the question), and writes `dataset.jsonl`
   (`query` + `context`) into the agent folder.
2. `run_eval.py` creates **one** evaluation (built-in `builtin.groundedness`
   grader) and then **one run per model**. For each model, Foundry regenerates
   the answer from the same `context` + `query`, then grades how grounded that
   answer is. All runs sit under the same evaluation so the portal shows them as
   a comparison table. Runs use the `azure_ai_target_completions` data source
   with an `azure_ai_model` target, so the portal's **Target** column shows the
   model that produced each run's answers.

## Prerequisites

- `az login` with access to the project's resource group and Foundry project.
- The Foundry project must have Application Insights connected (it is, via
  `infra/modules/foundry.bicep`).
- `azd env` provisioned (the scripts read `FOUNDRY_PROJECT_ENDPOINT`,
  `AZURE_RESOURCE_GROUP`, etc. from `azd env get-values`).

## Usage

```powershell
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt

# 1. Build the dataset from Foundry traces for an agent folder
.\.venv\Scripts\python.exe fetch_traces.py aurelia-motors --days 90

# 2. Run the multi-model groundedness comparison (models come from agent.json)
.\.venv\Scripts\python.exe run_eval.py aurelia-motors

# ...or override the models on the command line
.\.venv\Scripts\python.exe run_eval.py aurelia-motors --models gpt-4o-mini,gpt-5.4
```

`run_eval.py` prints a comparison table and a **View in the new Foundry portal**
link to the evaluation (under Build → Evaluations).

## Add a new agent

1. `mkdir agents/<agent-name>` and add an `agent.json` (copy aurelia-motors').
2. `python fetch_traces.py <agent-name>`
3. `python run_eval.py <agent-name>`

## Notes

- The evaluation is created via the Foundry **Evals API**, so it appears in the
  **new** Foundry portal. (The older `azure.ai.evaluation.evaluate()` SDK writes
  to the classic evaluation store, which is only visible with the *New Foundry*
  toggle off.)
- The Foundry account has key auth disabled; everything authenticates with Entra
  ID. `run_eval.py` clears any stray `AZURE_OPENAI_API_KEY` / `OPENAI_API_KEY`
  from the environment so the openai client does not attach an api-key header.
- Pass `--workspace <law-guid>` to `fetch_traces.py` to skip Log Analytics
  workspace auto-discovery, or `--keep-empty-context` to include turns that had
  no grounding context.
